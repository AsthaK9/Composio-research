01 — The patterns

Insight over raw table

Four things stand out once the 100 rows are clustered. This is the headline; the full 100-row register is further down for anyone who wants to check a specific app.

Auth OAuth2 dominates

62 of 100 apps use OAuth2, 28 use a flat API key/token, the rest are Basic auth, no-auth CLIs, or unclear. OAuth2 winning doesn't mean "easy" — see Marketing/Ads below.

Easiest categories Dev tools & PM = 100%

Developer/Infra (Vercel, GitHub, Supabase…) and Productivity (Notion, Linear, Asana…) are 100% self-serve. These categories are built by developers, for developers — the gate basically doesn't exist.

Hardest category Finance = 50% self-serve

Fintech is the worst-scoring category: only 4 of 10 apps are ready today. Corporate cards (Brex, Ramp) and data platforms (PitchBook) gate on "are you already our customer," not on auth complexity.

Most common blocker "Approve my app," not "gate my access"

Google Ads, Meta Ads, LinkedIn Ads, WhatsApp and Threads are all technically self-serve OAuth2 — but production use needs a developer-token or app-review step. That's a queue, not a wall, and it's the single most common workaround pattern (7 apps).

02 — By category

Where the easy wins are vs. where outreach is needed

Ranked by share of self-serve access, highest first. Developer tools and productivity apps need zero human involvement to start building; Finance and AI/Research need either a sales conversation or patience with a fast-moving, still-undocumented API.

04 — The agent

A two-pass pipeline, not a hand-typed table

One model call per app, twice: once cold (baseline), once with a real web-search tool attached and instructions to cite the actual docs page it read (verified). The gap between those two passes is the accuracy story — see Verification below.

STEP 1

Seed list

100 apps in, id + name + category. apps_seed.csv.

STEP 2

Baseline pass

Model answers the full schema from training knowledge only. Fast, free, right about a third of the time on anything obscure.

STEP 3

Verified pass

Same schema, web_search tool attached, must cite a URL it found this turn. This is what actually produced apps.json.

STEP 4

Diff + human review

Field-level diff between passes flags what changed; a human spot-checks the cited URLs and writes the honest exceptions.

Where it needed a human

Defining the taxonomy

The agent classifies into self_serve / self_serve_paid / approval / partner_gated — but a human drew those category boundaries and decided, e.g., that "OAuth2 + app review" is approval, not self_serve, even though the OAuth handshake itself is instant.

Catching a name collision

The first search for "Pylon" returned an unrelated open-source GraphQL framework, not the support tool at usepylon.com. The agent would have shipped a confident, wrong answer about the wrong product — a human noticed the category mismatch and re-ran the query.

Deciding when "not found" is the finding

Pumble, Waterfall.io, Paygent Connect and iPayX have no clear public developer docs. A human decided these should be reported as honest gaps, not re-queried forever or quietly guessed into a plausible-looking row.

Final accuracy scoring

The pass-1-vs-pass-2 diff is mechanical; deciding which field-level changes counted as a real "miss" vs. a harmless rewording, and writing up why each one happened, was a human editorial pass.

06 — Proof & repro

Run it yourself

The dataset above isn't a one-off — research_agent.py reproduces the whole pipeline against any app list, and swaps cleanly to Composio's own MCP tools for the search step.

Reproduce the verified pass

pip install anthropic --break-system-packages
export ANTHROPIC_API_KEY=sk-ant-...

python agent/research_agent.py \
  --input agent/apps_seed.csv \
  --pass verified \
  --out apps.json

One model call per app, web_search attached, must cite a live URL. Swap the tool block for a Composio MCP web-search action and nothing else changes — see README.
